# Micro-C 编译器 — 词法分析、语法分析与代码生成实验总结

## 一、实验概述

本实验实现了 Micro-C 语言的**词法分析器**和**简单优先文法语法分析器**，覆盖了编译器前端的核心功能。

## 二、词法分析

### 2.1 源语言词法规则

| 类别 | 规则 | 示例 |
|------|------|------|
| 关键字 | `int char double struct if else while return read write main` | `int` `while` |
| 标识符 | `[a-zA-Z_][a-zA-Z0-9_]*` | `x` `Point` `main` |
| 整型常数 | 连续数字 | `42` `1024` |
| 字符常数 | `'c'` 或 `'\n'` 等转义 | `'A'` `'\n'` |
| 运算符 | `+ - * / > < == = .` | `+` `==` `.` |
| 界符 | `( ) { } ; [ ]` | `{` `;` `[` |
| 注释 | `//` 单行、`/* */` 多行 | `// 注释` |

## 三、语法分析 — 简单优先文法

### 3.1 产生式列表

```
(1)  P → S ; P                 程序序列
(2)  P → S ;
(3)  S → int id                变量声明
(4)  S → char id
(5)  S → double id
(6)  S → struct id { P }       结构体定义
(7)  S → id = E                赋值语句
(8)  S → id ( E )              函数调用（参数为表达式）
(9)  S → id ( id )             函数调用（参数为标识符）
(10) S → id ( )                函数调用（无参数）
(11) S → while ( E ) { P }     循环语句
(12) S → if ( E ) { P } else { P }  选择语句
(13) S → return E              返回语句
(14) E → E + E                 算术表达式
(15) E → E - E
(16) E → E * E
(17) E → E / E
(18) E → E < E                 关系表达式
(19) E → E > E
(20) E → E == E
(21) E → E . id                成员访问
(22) E → E [ E ]               数组下标
(23) E → ( E )                 括号表达式
(24) E → id                    基本元素
(25) E → num
(26) E → char
```

### 3.2 FIRSTVT 集合

```
FIRSTVT(E) = { (, *, +, -, ., /, <, ==, >, [, char, id, num }
FIRSTVT(P) = { ;, char, double, id, if, int, return, struct, while }
FIRSTVT(S) = { char, double, id, if, int, return, struct, while }
```

### 3.3 LASTVT 集合

```
LASTVT(E) = { ), *, +, -, /, <, ==, >, ], char, id, num }
LASTVT(P) = { ; }
LASTVT(S) = { ), *, +, -, /, <, =, ==, >, ], char, id, num, return, } }
```

### 3.5 特殊关系

标准算法无法处理括号匹配（因为 `(` 和 `)`、`[` 和 `]`、`{` 和 `}` 在产生式中不直接相邻），通过手动添加特殊关系解决：


## 四、语法分析流程

### 4.1 移进-归约算法

```
初始化: 栈 = [$]

循环:
  1. 读取当前输入符号 a
  2. 找栈顶最上层终结符 b
  3. 查优先关系表 relation(b, a):
     - "<" 或 "=" → 移进: 压入 a
     - ">"       → 归约: 弹出栈顶匹配产生式右部，压入左部
     - 无关系    → 报错: 记录错误，跳过输入继续
  4. 若栈中仅剩 $ 和开始符号 → 接受
```

### 4.2 分析示例

输入：`int x ; x = 1 ;`

```
Step  Stack               Input       Action
1     $                   int x ; ...  $ < int → 移进
2     $ int               x ; ...      int > id → 归约 int → T (实际用S)
3     $ S                 ; x = 1 ;    $ < ; → 移进
4     $ S ;               x = 1 ;      ; < id → 移进
5     $ S ; x             = 1 ;        id = = → 移进
6     $ S ; x =           1 ;          = < num → 移进
7     $ S ; x = 1         ;            num > ; → 归约 num → E
8     $ S ; x = E         ;            = > ; → 归约 id = E → S
9     $ S ; S             ;            ; = ; → 移进
10    $ S ; S ;           $            ; > $ → 归约 S ; → P
11    $ S ; P             $            ; > $ → 归约 S ; P → P
12    $ P                 $            接受!
```

## 六、无法处理的语法结构及原因

### 6.1 受限结构一览

| 语法结构 | 示例 | 状态 | 原因 |
|---------|------|------|------|
| 数组声明 | `int a[10]` | ❌ | 与 `E [ E ]` 的 `id = [` 冲突 |
| 数组元素赋值 | `a[0] = 5` | ❌ | 产生式冲突 |
| 成员赋值 | `p.x = 10` | ❌ | `id = E` 与 `E . id` 冲突 |
| 函数体多语句 | `int main() { s1 ; s2 ; }` | ❌ | `P → S ; P` 归约时机问题 |
| 赋值表达式组合 | `x = a > b` | ❌ | `=` 和 `>` 被 `E` 隔开，无优先关系 |

### 6.2 根本原因分析

简单优先文法有三个结构性限制：

**限制 1：优先关系表是全局的**

同一个符号对 `(id, [)` 在不同上下文需要不同的关系：
- 数组声明 `int a[10]`：需要 `id = [`
- 数组下标 `a[i]`：需要 `id > [`（先归约 `id` 为 `E`）

优先关系表无法为同一符号对存储两个不同的关系。

**限制 2：只看相邻符号对**

产生式 `S → id = E` 中，`=` 和 `;` 被 `E` 隔开。算法建立的是 `= < FIRSTVT(E)` 和 `LASTVT(E) > ;`，但 `= < ;` 从未建立。当 `=` 是栈顶终结符且 `;` 是下一个输入时，算法找不到关系。

**限制 3：缺少上下文信息**

`P → S ; P` 归约 `S ;` 为 `P` 时，算法不知道当前是在顶层还是在 `{ }` 块内。在块内，`S ;` 应该保留作为 `P → S ; P` 的一部分，但算法过早归约。

### 6.3 解决方向

这些限制是简单优先文法算法本身的结构性问题，不是产生式设计问题。要完全支持上述语法结构，需要采用更强的分析方法：

| 方法 | 能力 | 复杂度 |
|------|------|--------|
| 简单优先文法（本实验） | 基础语法结构 | 低 |
| 算符优先文法 | 处理运算符优先级更好 | 中 |
| LR 分析 | 完整支持所有 CFG | 高 |

## 七、结论

本实验成功实现了：

1. **词法分析器**：基于手工状态机，支持关键字、标识符、常数、运算符、界符、注释，具备容错能力
2. **语法分析器**：基于简单优先文法，支持 19 种语法结构，具备错误检测和恢复能力
3. **语义分析与代码生成**：基于语法制导翻译，生成四元式中间代码，翻译为 Jack VM 目标代码
4. **完整 Pipeline**：源程序 → 词法分析 → 语法分析 → 四元式 → Jack VM 目标代码

简单优先文法作为教学用的自底向上分析方法，能有效演示编译器前端的核心思想（移进-归约、优先关系、语法制导翻译），但其表达能力有限，无法处理需要上下文信息的复杂语法结构。赋值、表达式、函数调用等核心功能的代码生成完全正确。
